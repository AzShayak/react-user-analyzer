Purpose

Thank you for your interest in joining our team. We need to evaluate

  • how proficient the candidates are at building large-scale software systems
  • how attentive the candidates are to detail

Completing this take-home question will give us the data points we need to make this evaluation. Thank you for the time you put into building the solution.

We strongly encourage you to read in full and understand the Chaldal Engineering Standards document engineering-standards.html, and adjust your problem solving and coding approach to match the criteria we'll be evaluating you on.

Task

Build a mobile web application using React to analyze app usage data for a hypothetical menu planning calendar app. On any day, a user can plan multiple meals, and each meal may have multiple dishes. The program you build will analyze how engaged users are with the app.

Given a specific date range, your analyzer should filter users into three types

  • Active: users who had at least 5 meals in the specified period
  • Superactive: users who had more than 10 meals in the specified period
  • Bored: Users who were “active” in the preceding period, but not “active” in the specified period.

In the ./data folder, you will find a bunch of files in the form userId.json. They contain app data for a bunch of users. The structure should be fairly obvious upon examination.

In the design folder, you will find the following three design files to aid you in your implementation.

  • 00-select-filter-option.png shows filters used to generate the user list.
  • 01-show-user-grid.png shows the searchable list of users based on the filter selected
  • 02-edit-filter-options.png shows UI to edit filter and regenerate user list

Example

Selecting a date range from 2016-09-01 to 2016-09-08 and Status as "active" should generate a list of “active” users during the specified period.

Guidelines

We do not expect you to spend more than 6 hours implementing this solution. If after spending 2-3 hours on this assignment you realize you that you might not be able complete the entire solution within 6 hours, we suggest you implement a simplified version of the design and send it in.

Review the engineering standard document to understand how your solution will be evaluated. The data layer, business logic and user interface elements of your code should be separated.

User Interface

  • Your solution should work well on mobile web.
  • Pixel perfection is not necessary, but we will critically evaluate the neatness of the design and how user friendly it behaves.
  • Responsive design to fit desktop web is not required but will be a bonus

Assumptions

You may make reasonable assumptions where you see fit, or where these requirements are unclear. In such cases, you do not need to check with us on the validity of the assumptions; simply state what they are in a “README.md” under the project root.

Please do not send instructions in your response e-mail. Information sent in the email body will not be reviewed by evaluators.

Style

There are many ways to write such a program. When choosing your style, imagine that you are building a system that will evolve to handling many more features in the very near future, so structure your code in a way that would support such a future.

Submission

Please submit the code, data and whatever else you deem necessary to help us reach our stated evaluation goals in a single compressed archive.

Do not include any built binaries or executables within your submission, as it may get filtered by anti-virus software and will not reach us.

You may communicate any notes to the evaluators using README.md. Any notes you may add in emails will not reach the evaluators, so please ensure you only use README.md for such purposes.
